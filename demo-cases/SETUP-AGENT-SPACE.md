# Agent Space Setup Checklist (manual, console-only)

This file is the **human runbook** for the steps that cannot be automated
through Terraform or CLI. Everything else (infra, faults, app, webhook
bridge) is scripted.

## Prerequisites

Before you start, the following must already be true:

- [ ] Terraform infra applied in both China accounts (`bjs1-stack` and
      `china-stack` under `demo-cases/infra/`).
- [ ] Existing project's `aws-cn` and `aws-cn-2` MCP servers are running on
      EKS in `us-east-1` and reachable from the DevOps Agent space via the
      VPC Lattice Private Connection that this repo's parent project set up.
- [ ] You have admin access to the AWS account that hosts the DevOps Agent
      space (this is **not** one of the China accounts — DevOps Agent does
      not run in `aws-cn` partition).
- [ ] You have a Slack workspace where you can install third-party apps.
- [ ] You have a GitHub account with admin permission on the repo
      `bjs-todo-api` (the demo app — owned by `ychchen` on GitHub).

## Step 1 — Create or reuse the Agent Space

If you already have the Agent Space from blogs 01-02 with the two China MCP
servers connected, **reuse it**. Otherwise:

1. Open the DevOps Agent console in the supported region (`us-west-2` or
   `us-east-1` — pick the one your existing MCP setup uses).
2. Settings → Agent Spaces → Create.
3. Name it `china-demo-cases-2026-05` (descriptive — easy to find later).
4. Attach the two existing China account MCP servers (`aws-cn`, `aws-cn-2`).
   These are not real AWS accounts in the Agent Space sense — they are MCP
   servers reachable through the Private Connection. Confirm both show
   "Connected" status.
5. **Skip** native AWS account integration for the China accounts. The
   agent does not support `aws-cn` partition for native account access.
   All China-region work goes through the MCP servers.

## Step 2 — Generate the webhook URL and secret

The webhook bridge Lambda (`demo-cases/infra/webhook-bridge/`) cannot create
this; it must be generated by hand and the URL+secret saved into SSM.

1. Agent Space → Capabilities → Webhook → Generate.
2. **Copy the URL and the secret immediately** — the secret is shown once
   and never again.
3. Decide where to store them. Recommended: SSM Parameter Store in **both**
   `ychchen-bjs1` and `ychchen-china` (each China account needs its own
   copies because the bridge Lambda runs locally in each account).

   ```bash
   unset AWS_PROFILE AWS_REGION

   # bjs1
   aws --profile ychchen-bjs1 ssm put-parameter \
       --name /devops-agent/webhook-url \
       --type String \
       --value "https://<copied-url>" \
       --overwrite

   aws --profile ychchen-bjs1 ssm put-parameter \
       --name /devops-agent/webhook-secret \
       --type SecureString \
       --value "<copied-secret>" \
       --overwrite

   # china — same two commands with --profile ychchen-china
   ```

4. Verify: the bridge Lambda terraform module (`webhook-bridge/terraform/`)
   reads these parameter names. If you change names here, update the module
   variables.

## Step 3 — Connect GitHub

The demo app `bjs-todo-api` is in a GitHub repo. The agent needs read-only
access to it for cases C2, C7, C9 (deploy-to-incident time correlation
and agent-ready spec generation).

1. Agent Space → Capabilities → Pipeline → Add → GitHub.
2. Complete the OAuth flow. Choose **specific repos**, not all repos.
3. Select only `bjs-todo-api` (and any other demo repos you've created).
4. Confirm the integration shows "Connected" with at least 1 repo bound.

## Step 4 — Connect Slack

For C1 (autonomous webhook investigation), the agent needs to publish
investigation timelines and RCA reports back to a Slack channel.

1. Slack workspace → Apps → install "AWS DevOps Agent" (you may see a
   "not approved by Slack" badge — that is currently expected; it works).
2. Agent Space → Settings → Communications → Register → complete the
   Slack OAuth flow.
3. Create a channel `#bjs-web-incidents` (private is fine).
4. In Slack, run `/invite @AWS DevOps Agent` in that channel.
5. Back in the Agent Space console: Communications → Add integration →
   Slack → paste the channel ID (in Slack, right-click the channel name
   → View channel details → copy ID at the bottom).

## Step 5 — Upload custom skills

Eight skills already exist in `skills/` (foundation, 3 analytical,
4 incident-pipeline). One new skill is added for these demo cases:
`cn-partition-arn-routing` (drives case C5 "agent gets it wrong → skill
saves the day").

For each skill (existing or new):

1. Agent Space → Skills → Add skill → Upload.
2. Upload the corresponding `.zip` file from `skills/`. The console will
   parse the SKILL.md frontmatter and pre-fill `name` and `description`.
3. **Set Agent Type assignments** as documented in
   `blog/03-skills-in-action.md` § 5:
   - `china-region-multi-account-routing` → Generic (shared)
   - `cross-account-inventory-compare` → Generic + On-demand
   - `cross-account-cost-attribution` → Generic + On-demand
   - `cross-account-security-posture-check` → Generic + On-demand
   - `china-account-prevention-checks` → **Evaluation only**
   - `china-incident-triage` → **Incident Triage only**
   - `china-incident-rca` → **Incident RCA only**
   - `china-incident-mitigation` → **Incident Mitigation only**
   - `cn-partition-arn-routing` → Generic + Incident RCA + Incident Mitigation
     (the partition bug shows up both in ad-hoc queries and during
     pipeline investigations)

4. After upload, validate the skill list shows all 9 (8 existing + 1 new)
   plus the 2 built-in core skills (`Tool use best practices`,
   `understanding-agent-space`). Total visible: 11.

## Step 6 — Smoke test the wiring

Quick checks before running the demo cases:

- [ ] Generic agent reachable: open Operator Web App, ask "list MCP
      servers connected to this Agent Space" — expect both China MCPs.
- [ ] Webhook reachable from bridge Lambda: in `ychchen-bjs1`, manually
      invoke the deployed bridge Lambda with the test SNS payload from
      `webhook-bridge/tests/sample_event.json`. Expect a 200 response.
- [ ] GitHub: ask the agent "what was the latest commit on bjs-todo-api"
      — expect a real commit hash.
- [ ] Slack: in the Agent Space console, run any short investigation;
      confirm a message appears in `#bjs-web-incidents`.
- [ ] Skills picker: ask the Generic agent "对比 aws-cn 和 aws-cn-2 的 VPC
      CIDR" — confirm the skill use panel shows
      `china-region-multi-account-routing` (and ideally
      `cross-account-inventory-compare`) was activated.

If any of these fail, fix before starting Phase 4 case execution. The
demo cases assume all five smoke tests pass.

## Step 7 — Pricing and trial state

- DevOps Agent has a 2-month free trial per AWS account; verify your
  account is still inside it.
- Trial includes 20 hours/month investigation, 15 hours evaluation,
  20 hours on-demand. Demo budget assumes ~2.5 hours of investigation
  total — comfortably within free tier if trial is active.
- Outside the trial: $0.0083/agent-second ≈ ¥217/hour. Budget allowance
  in the design doc assumes worst case (no trial).
- **Set a CloudWatch billing alarm** on the Agent Space's host account
  if you are paying. There is no DevOps-Agent-side cost cap.

## Step 8 — Teardown checklist (Phase 5)

When finished:

- [ ] Delete the webhook in Agent Space → Capabilities → Webhook → Delete.
      (If you keep the URL, anyone with the secret can still trigger
      investigations against your space.)
- [ ] Disconnect GitHub integration (the GitHub app stays installed in
      your GitHub account; uninstall it there separately).
- [ ] Disconnect Slack integration (the Slack app stays in workspace;
      remove from the channel and uninstall if you no longer need it).
- [ ] Delete uploaded custom skills if you don't want to retain them.
- [ ] `terraform destroy` in both China accounts.
- [ ] Delete the SSM parameters for the webhook URL and secret.
- [ ] Run `demo-cases/faults/recover-all.sh` first if any fault is still
      injected — destroy may fail if a fault left resources in unexpected
      states.

## Known issues & gotchas

- **Skill description colon-space (`: `) bug** (from blog 03 § 4): if you
  edit any SKILL.md description directly in the console, avoid plain-scalar
  YAML strings that contain `: ` — they break parsing. The pre-uploaded
  zips in this repo have already been linted for this.
- **Agent Type misassignment**: setting all skills to Generic seems
  convenient but pollutes context for incident-only agents. Follow the
  table in Step 5 strictly.
- **Trial-versus-paid**: the on-demand hour bucket is shared across the
  whole Agent Space, not per-investigation. If you've been heavily using
  the space, the demo may push you into paid usage. Check
  Settings → Billing before starting.
- **China region fact**: even though the MCP servers expose China account
  resources, the Agent Space itself runs in a non-China region. Slack/GitHub
  webhooks therefore traverse the public internet and your VPN/firewall
  rules need to allow that.
